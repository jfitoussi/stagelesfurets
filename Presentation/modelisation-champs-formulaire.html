Programmation en binôme

Définition : une vieille méthode de développement Agile dans laquelle deux développeurs travaillent ensemble sur un même poste de travail.

Pour 15% d'augmentation du temps de développement

(si vous me croyez pas demandez a Mme Williams...)

Avantages pour un développeur

  • améliore la satisfaction des développeurs
  • améliore les capacités de résolution de problème
  • permet de découvrir de nouveaux idiomes
  • rend les tâches difficiles plus accessibles
  • rend les tâches monotones plus intéressantes
- satisfaction : plus agréable de travailler a deux / plus facile - résolution : plus de solutions envisagées - idiômes : nouvelles fonctions langages, env. développement - tâches difficiles : donne du courage d'être a deux - tâches monotones : donne la force de faire les refactorings (XP)

Avantages pour une société

  • améliore la cohésion d'équipe
  • améliore les problématiques de recrutement
  • augmente le "nombre bus"
  • les inspections sont effectuées JIT
  • permet de faire de la revue de code gratuitement
- cohésion : plus de communication, développeurs plus proches - recrutement : intégrations plus faciles, départs plus faciles - nombre bus : nombre de départ qu'il faut pour rendre le projet a risque - inspections : détections des anomalies très tôt - revue code : anectdote sur les 2x plus de temps passé

Avantages pour un produit

  • améliore la qualité du design
  • diminue les coûts de productions
  • moins de chance de faire la solution "sale"
- design : phase la plus importante d'être en paire ! analyse tôt - coûts : détection plus rapides anomalie, moins maintenance - hacky : la pression sociale c'est fort et motivateur

Le développement...

... est souvent appris comme une activité solo

- a l'école : cours magistraux, TD seuls, pas de système de versionnement, pas de conférences, pas de code review, pas de pair prog, ect. - ça commence mais on y est pas encore

Une recette simple comme la poutine !

0Créer un environnement de développement

- penser a tout : claviers accessibles, OS, outils de dev... - rien d'exotique ! - peut-être différent par paire, mais s'assurer quelles peuvent tourner facilement

avec des outils accessibles...

1Sélectionnez correctement votre paire

pleins de combinaisons possible : - expert / junior, junior / junior, etc. (en âge ou sur un code en particulier) - pleins de théories la-dessus : pas important ! - définir un but et s'y tenir : intégrer un nouveau, améliorer le code, etc.

2Sélectionnez une tâche assez définie

...mais pas trop !

- difficile de choisir une paire s'il n'y a pas de tâche définie (on se rappelle du but) - pas trop définie parce que la partie la plus importante c'est : le design !

3Faire confiance en son partenaire et le supporter

- écouter et aller au fond des choses

4Parler beaucoup

"qu'est-ce que tu penses de..."

"comment ferais-tu pour..."

"je vais tester comme ça..."

- évite l'exclusion - évite la gêne

5Se synchroniser souvent et chercher l'approbation

- donne une structure a la session : "on en est la..." - évite les différences de phase - évite d'oublier des trucs

6Célébrer

- veuve cliquot seulement

7Varier les paires, varier le temps de travail

- paires : pour la raison ici-haut - temps : très éprouvant en fonction tâche (parfois 2 heures suffisent)

8Tourner souvent

- exemple : ping-pong development

9Faire des rétros sur la pratique

- encore une fois, histoire de communication

10Planifier les paires dans le temps

- si on planifie les tâches, alors on planifie les paires, ça va ensemble - si agile / scrum / les définir lors du planning poker

11Donner un temps de "jello"

- expliquer le jello - un de mes favoris : pas facile avec tout le monde, parfois 1 minute de jello, parfois 1 an - au moins quelques heure, comme tout nouveau process attendez vous a ce que ce soit pénible avant d'être efficace
